# Technical Reference — U by Moen Provisioning Protocol

This document covers the provisioning protocol internals, API tables, and legacy firmware details. For normal usage see [README.md](README.md).

> **Legal note:** This protocol was documented by analyzing network traffic generated by the official U by Moen app communicating with hardware the author owns. This is standard interoperability research permitted under the DMCA § 1201(f) exemption and consistent with the Computer Fraud and Abuse Act, as it involves only hardware and accounts owned by the researcher. No Moen servers were accessed beyond normal authenticated API calls any legitimate user would make.

## How Provisioning Works

The provisioning flow has two phases: **cloud authentication** and **local controller setup**.

### Phase 1 — Cloud Authentication

```
GET https://www.moen-iot.com/v2/authenticate?email=...&password=...
Accept: application/json

Response: {"token": "XXXXXXXXXXXXXXXX"}
```

This token identifies your Moen account and is cached in `moen_config.json`. It is passed to the controller during provisioning so it knows which cloud account to register with. The token is long-lived — you only need to fetch it once unless you change your password.

### Phase 2 — Local Controller Setup (on Moen AP)

Once connected to the Moen AP (`192.168.10.x`), the provisioning flow has 4 steps.

#### Step 1 — Get session key

```
GET http://192.168.10.1/v1/prov/session
Auth-Hash: sha256(paddedPIN + ":" + serial + ":" + paddedPIN)

Response: {"session_key": "<base64-encoded RSA public key>"}
```

The `Auth-Hash` proves physical presence — only someone standing at the shower can read the PIN. This prevents remote attackers from hijacking provisioning.

**Computing the Auth-Hash:**

```bash
PIN="AB12"
SERIAL="7654321"
PADDED=$(printf "%-16s" "$PIN" | tr ' ' '0')  # → AB12000000000000
echo -n "${PADDED}:${SERIAL}:${PADDED}" | sha256sum
# 092758d920ac236c47c4c6e77f282ca1f61142cb574d40ca316e0fadc55aa50c
```

#### Step 2 — Encrypt credentials

The session key is an RSA-2048 public key. Three fields are encrypted before sending:
- `user_token` — your Moen account token
- `ssid` — home WiFi SSID
- `password` — home WiFi password

Each field is encrypted as: `RSA_PKCS1v15_encrypt(field + ":" + SHA512(field))`

The SHA-512 hash appended to each field acts as an integrity check — the controller verifies it matches after decrypting.

**Encrypting a field:**

```bash
FIELD="MyHomeWiFi"
SHA512=$(echo -n "$FIELD" | sha512sum | awk '{print $1}')
PAYLOAD="${FIELD}:${SHA512}"

# Fetch and decode the session key first:
SESSION_KEY="<value from /v1/prov/session>"
echo "$SESSION_KEY" | base64 -d > session_key.pem

# Encrypt:
echo -n "$PAYLOAD" | openssl rsautl -encrypt -pkcs -pubin -inkey session_key.pem | base64
```

> RSA-2048 with PKCS#1 v1.5 can encrypt up to 245 bytes. The payload is always `len(field) + 1 + 128` bytes (128 hex chars from SHA-512). WiFi passwords max 63 chars — well within the limit.

#### Step 3 — Send registration (raw TCP)

The app uses a raw TCP socket rather than a standard HTTP library to avoid CORS preflight requests, which the controller does not support.

```bash
AUTH_HASH="092758d920ac236c47c4c6e77f282ca1f61142cb574d40ca316e0fadc55aa50c"
TIMESTAMP=$(date +%s%3N | head -c -3)
ENC_TOKEN="<base64 from Step 2>"
ENC_SSID="<base64 from Step 2>"
ENC_PASS="<base64 from Step 2>"

BODY=$(printf '{"user_token":"%s","shower_name":"Master Shower","ssid":"%s","password":"%s","api_server":"https://www.moen-iot.com"}' \
  "$ENC_TOKEN" "$ENC_SSID" "$ENC_PASS")

CONTENT_LENGTH=${#BODY}

REQUEST=$(printf "POST /v2/prov/registration HTTP/1.1\r\nAuth-Hash: %s\r\nTimestamp: %s\r\nContent-Type: application/json\r\nAccept: */*\r\nContent-Length: %d\r\nHost: 192.168.10.1\r\nConnection: Keep-Alive\r\n\r\n%s" \
  "$AUTH_HASH" "$TIMESTAMP" "$CONTENT_LENGTH" "$BODY")

# Linux:
echo -e "$REQUEST" | nc -q 3 192.168.10.1 80
# Mac:
echo -e "$REQUEST" | nc -G 3 192.168.10.1 80
```

Expected response on success:
```
HTTP/1.1 200 OK
Server: Marvell-WM
Connection: close
Content-Length: 0
```

#### Step 4 — Poll for completion

```
GET http://192.168.10.1/v1/prov/status
Auth-Hash: <same as above>

Response (connecting): {"status": "connecting", "progress": "connecting to network"}
Response (success):    {"status": "connected"}
Response (failure):    {"status": "error", "progress": "connecting to network failed"}
```

#### PIN Details

The PIN is generated fresh each provisioning session and proves physical presence. It is padded to 16 characters before use:

```
padded_pin = pin.upper().ljust(16, '0')
# e.g. "AB12" → "AB12000000000000"
```

## Cloud API

Base URL: `https://www.moen-iot.com`

All authenticated endpoints require a `User-Token: <user_token>` header.

| Endpoint | Auth | Description |
|----------|------|-------------|
| `GET /v2/authenticate?email=&password=` | none | Login, returns `{"token":"..."}` |
| `GET /v2/credentials?user_token=&serial_number=` | none | Get Pusher app_key, cluster, channel |
| `POST /v2/pusher-auth?user_token=&serial_number=` | none | Authenticate Pusher private channel |
| `GET /v5/showers/{serial}` | User-Token | Get shower details including `temperature_units` |
| `PATCH /v4/showers/{serial}` | User-Token | Update shower settings/presets |

## Pusher Cloud Control (firmware 3.x with `hmi_supports_pusher`)

Firmware 3.x controllers register the `hmi_supports_pusher` capability with the cloud. These controllers do **not** register local HTTP control routes — all control goes through Pusher WebSocket relay.

### Connection flow

1. `GET /v2/credentials` → `{"app_key": "...", "cluster": "us2", "channel": "<uuid>"}`
2. WebSocket connect to `wss://ws-{cluster}.pusher.com/app/{app_key}?protocol=7`
3. Receive `pusher:connection_established` → extract `socket_id`
4. `POST /v2/pusher-auth` (query: `user_token`, `serial_number`; body: `channel_name`, `socket_id`) → auth string
5. Send `pusher:subscribe` with `channel: "private-{channel}"` and `auth`
6. Send control events on the authenticated channel

### Events

| Event | Direction | Description |
|-------|-----------|-------------|
| `client-state-desired` | app → controller | Control actions (on/off/temp/preset) |
| `client-command` | app → controller | RPC calls (e.g. `do_shower_report`) |
| `client-state-reported` | controller → app | Response to `do_shower_report` |
| `client-command-resp` | controller → app | RPC response |

### Control actions (`client-state-desired`, type `"control"`)

```json
{"type": "control", "data": {"action": "shower_on",          "params": {}}}
{"type": "control", "data": {"action": "shower_off"}}
{"type": "control", "data": {"action": "temperature_set",    "params": {"target_temperature": 101}}}
{"type": "control", "data": {"action": "shower_on",          "params": {"preset": 1}}}
{"type": "control", "data": {"action": "outlets_set",        "params": {"outlets": [{"position": 1, "active": true}]}}}
```

### Settings (`client-state-desired`, type `"settings"`)

Used to update controller-level settings rather than shower state. The `data` field is the settings object directly (not wrapped in `action`/`params`).

```json
{"type": "settings", "data": {"homekit_enable": true}}
{"type": "settings", "data": {"homekit_enable": false}}
{"type": "settings", "data": {"max_temperature": 110, "use_fahrenheit": true, "display_brightness": 2}}
```

Available settings fields (from `HMISettings.java`):

| Field | Type | Description |
|-------|------|-------------|
| `homekit_enable` | bool | Enable/disable the controller's HomeKit accessory server |
| `max_temperature` | int | Maximum allowed temperature (°F) |
| `use_fahrenheit` | bool | Display units on controller screen |
| `display_brightness` | int | Screen brightness level |
| `idle_display_off` | bool | Turn off screen when idle |
| `time_zone` | int | Timezone offset index |
| `language` | int | UI language index |

> `target_temperature` is always in **Fahrenheit** — the controller stores all temperatures internally as °F. `moen_control.py` handles conversion automatically based on your cloud account's `temperature_units` setting.

### Temperature units

## Controller Local API

### Before Provisioning — Moen AP mode (`http://192.168.10.1`)

Available only while the controller broadcasts its own AP (`Moen_XXXXXX`).

| Endpoint | Method | Auth-Hash | Description |
|----------|--------|-----------|-------------|
| `/v2/ping` | GET | none | Check controller reachable |
| `/v1/prov/networks` | GET | none | List nearby WiFi networks |
| `/v1/prov/session` | GET | `sha256(paddedPIN:serial:paddedPIN)` | Get RSA session key |
| `/v2/prov/registration` | POST (raw TCP) | `sha256(paddedPIN:serial:paddedPIN)` | Send encrypted WiFi credentials |
| `/v1/prov/status` | GET | `sha256(paddedPIN:serial:paddedPIN)` | Poll provisioning status |
| `/v1/prov/acknowledge` | POST | `sha256(paddedPIN:serial:paddedPIN)` | Acknowledge (older firmware path) |

### After Provisioning — Home WiFi mode (legacy / pre-Pusher firmware only)

> ⚠️ **Firmware 3.x controllers with `hmi_supports_pusher` do NOT register these routes.** All `/v1/shower` requests return `File /path not_found`. Use `moen_control.py` (Pusher) or `moen_local.py` with HomeKit for offline control.

For **older firmware** (pre-Pusher), once provisioned the controller is reachable at its home network IP (discoverable via mDNS as `moen-dolphin._http._tcp.local.`).

**mDNS discovery:** resolve hostname `moen-dolphin.local` — on Linux this requires Avahi, on macOS it works natively. TXT record includes `sn=<serial>` for identification.

Auth-Hash for these endpoints uses the **shower token** (not the PIN):
```
sha256(showerToken + ":" + serial + ":" + showerToken)
```

> The `showerToken` is different from your cloud `user_token`. It is returned by `GET /v5/showers/{serial}` as the `token` field and stored by `setup_moen.py` as `shower_token` in `moen_config.json`.

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/v1/shower` | GET | Get current shower state (encrypted response) |
| `/v1/shower` | POST | Set shower state — AES-128-CTR encrypted body |
| `/v1/refresh` | POST | Push updated settings to controller display |
| `config` | POST | Update controller config |

**AES-128-CTR encryption (applies to both POST body and GET response body):**

```
Key  = shower_token[:16]           (first 16 ASCII chars, used as-is as bytes)
IV   = sha256hex(timestamp)[:16]   (first 16 chars of the hex digest, as ASCII bytes)
Mode = AES/CTR/NoPadding
```

For **GET responses**, the `Timestamp` header in the HTTP response is the IV seed used for decryption (the controller encrypts the response using its own timestamp).

For **POST requests**, the client generates a timestamp, sends it in the `Timestamp` header, and uses the same value as the IV seed for encryption.

**Python implementation:**
```python
import hashlib
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes

def aes_ctr(data: bytes, timestamp: str, shower_token: str, encrypt: bool) -> bytes:
    key = shower_token[:16].encode()
    iv  = hashlib.sha256(timestamp.encode()).hexdigest()[:16].encode()
    c   = Cipher(algorithms.AES(key), modes.CTR(iv))
    op  = c.encryptor() if encrypt else c.decryptor()
    return op.update(data) + op.finalize()
```

**POST body schema** (`UpdateShowerRequest`, serialized as JSON before encryption):

| Field | Type | Notes |
|-------|------|-------|
| `current_mode` | string | `"adjusting"` (on), `"off"`, `"paused"` |
| `mode` | string | `"ready"` (on), `"off"` |
| `target_temperature` | int | °F |
| `active_preset` | int\|null | Preset number, null for manual |
| `outlets` | array | `[{"position": N, "active": bool}]` |

Minimal payloads are accepted — include only the fields being changed. `moen_local.py` does a GET-modify-POST cycle to preserve unchanged fields.

#### Fallback provisioning path (older firmware)

If the controller doesn't support `/v1/prov/session`, the app encrypts credentials with AES-CTR (PIN as key) and sends to `/v1/prov/registration`. `moen_provision.py` handles both paths automatically.
